### [上午] 優化 Cloudflare API 請求速率限制（Rate Limiting Overhaul）

#### 設計階段
- **分析現有問題**：`cf_smart_cache_check_rate_limit()` 固定時窗、硬編碼 1000 上限（非 1200）、所有呼叫者忽略回傳值、無退避重試、無 Purge API token bucket 感知、無請求隊列
- **研究 Cloudflare 官方文檔**：確認 Global API limit = 1200/5min，Purge API = token bucket 模型（Free: 5 req/min, bucket 25, max 100 ops/req），支援 `retry-after` header
- **產出完整生產級計劃**：滑動時窗 + Token Bucket + Adaptive Limit + Exponential Backoff with Jitter + Debounced Queue + HTTP Executor Retry + Admin UI Dashboard

#### 實作階段
- **新核心 Governor**：`cf_smart_cache_rate_governor()` — 滑動時窗計數、狀態機（normal/warning/critical/backoff）、自動調適限流（遇 429 降 10%，下限 600）
- **Token Bucket**：`cf_smart_cache_purge_bucket()` — 根據 Cloudflare 方案選擇參數、支援 burst
- **智慧退避**：`cf_smart_cache_backoff_delay()` — exponential 級數 [1,2,4,8,15] + ±20% jitter + `retry-after` header 感知
- **429 Response Handler**：`cf_smart_cache_handle_429_response()` — 調降 adapted_limit、track 連續 429、1 小時無 429 逐漸恢復
- **HTTP Executor**：`cf_smart_cache_http_request()` — 封裝所有 `wp_remote_*` 呼叫、3 次重試、讀 `retry-after` header、限流檢查
- **Debounced Purge Queue**：`cf_smart_cache_enqueue_purge()` + `cf_smart_cache_flush_purge_queue()` — 合併 2 秒內多個 purge 事件、去重、滿 100 URLs 強制 flush
- **修改所有 Caller**：`batch_purge()`、`execute_purge()`、`purge_all_cache()`、`fetch_zones()` — 全部改用 `cf_smart_cache_http_request()`、嚴格檢查限流回傳值
- **修改所有 Hook 端**：`on_status_change()`、`on_delete_post()`、`on_term_change()` — 改用 `cf_smart_cache_enqueue_purge()` 取代直接呼叫
- **Admin UI**：在 `display_cache_status()` 新增 Rate Limit 狀態區塊、新增 settings 欄位（max requests、retries、adaptive、plan type、batch size）
- **修正 Flaws**：`cf_smart_cache_check_rate_limit()` 日誌寫 1200 但判斷 1000 的不一致

### 主要決策
- **選擇滑動時窗而非固定時窗**：避免 5 分鐘邊界瞬間雙倍請求的已知問題
- **Token Bucket 用個別 transient**：與 global rate limit 分離管理，因為 Purge API 有其獨立的 token bucket 模型
- **Debounce queue 用 wp_schedule_single_event**：而非持續 polling，減少不必要的 transient 讀寫
- **保留舊函數 `cf_smart_cache_check_rate_limit()`**：內部改為呼叫新 governor，維持向後相容

## 遇到的問題和解決方案
- **問題**：舊函數 `cf_smart_cache_check_rate_limit()` 被多處呼叫但回傳值被忽略
  - **解決**：全部改寫為 `cf_smart_cache_http_request()`，回傳值統一為 response array 或 WP_Error
- **問題**：`cf_smart_cache_batch_purge()` 已有 sleep(1) 硬編碼在批次之間
  - **解決**：移除硬編碼 sleep，改用 governor 動態決策

### [下午] 執行 Phase 1：緩存機制核心優化

#### 1.1 廢棄舊式 purge0/1/2 系統
- **移除函數**：`cf_smart_cache_purge()`、`purge0()`、`purge1()`、`purge2()`、`post_transition()`

- **效果**：消除雙重清除（舊 header+hint + 新 API 清除 → 僅 API 清除）

#### 1.2 動態 TTL + stale directives
- **新增**：`cf_smart_cache_get_ttl()` 依據內容類型回傳差異化 TTL
  | 類型 | s-maxage | max-age | swr | s-error |
  |------|----------|---------|-----|---------|
  | 首頁 | 3600 | 1800 | 86400 | 604800 |
  | 文章/頁面 | 14400 | 7200 | 86400 | 604800 |
  | 歸檔 | 7200 | 3600 | 86400 | 604800 |
  | Feed | 1800 | 900 | 86400 | 604800 |
  | 其他 | 3600 | 1800 | 86400 | 604800 |
- **修改**：`set_edge_headers()` cacheable 分支簡化為單一路徑，動態產生 `Cache-Control` header
- **加入 filter**：`cf_smart_cache_ttl` 讓開發者自訂

- **雙層快取策略**：
  - Layer 1: `wp_cache_get/set`（per-request）避免同次請求多次 hook 重複查詢
  - Layer 2: `post_meta('_cf_smart_cache_purge_hash')`（cross-request）透過 hash 比較決定是否重新生成
- **新增**：`cf_smart_cache_purge_urls_hash()` — hash 包含 post_status/post_type/post_author + 所有 taxonomies term IDs
- **清理**：`on_delete_post()` 自動刪除 post meta cache
- **效果**：大型站點 taxonomies 查詢次數降低 >80%

### [下午 第二階段] 規劃 + 實作 Auto-Configuration Wizard

#### 背景
使用者不需手動設定 Cloudflare Page Rule / DNS Proxy / Edge Cache TTL，透過插件一鍵自動完成。

#### 關鍵發現（API 調研）
1. **Edge Cache TTL Zone Setting** 不支援 `"respect_origin"` 值（僅接受數值）。但 Cloudflare Free/Pro/Business 預設已啟用 Origin Cache Control，我們的 `Cache-Control: s-maxage=` header 本來就會被尊重 → **改檢查 `explicit_cache_control` 設定**
2. **Page Rule 只有最高 priority 的 rule 生效** → 必須在單條 rule 內合併多個 action
3. **Token 權限無法預先驗證** → 採 fail-and-tell 策略：執行 API 若 403 則顯示訊息
4. **Free 方案只有 3 條 Page Rule** → UI 顯示用量與剩餘數

#### 設計決策
- **Page Rule 最終 payload**：`cache_level=cache_everything` + `edge_cache_ttl=0`(Respect Origin) + `explicit_cache_control=on`
- **備份保留 3 版**：`cf_smart_cache_config_backups` option，FIFO 淘汰
- **回滾 ID 精確匹配**：紀錄我們建立/modified 的 rule ID
- **DNS Proxy 只改根域名**：不碰子域名
- **Apply 前顯示 diff**：讓用戶確認變更

### [下午 第四階段] 修復 Auto-Config API 認證遺失 Bug

#### 症狀
按 Apply 後三項全失敗：`page_rule` → "Invalid Page Rules response"、`explicit_cc` → "Failed to set explicit_cache_control"、`dns_proxy` → "Failed to fetch DNS records"

#### 根因
所有 auto-config 函數（`get_page_rules`、`apply_page_rule`、`get_zone_setting`、`apply_zone_setting`、`get_dns_records`、`apply_dns_proxy`、`delete_page_rule`、`backup_config`、`restore_backup`）呼叫 `cf_smart_cache_http_request()` 時**沒有傳入 `Authorization: Bearer` header**。對比之下，現有的 purge 函數全部有傳。

Cloudflare API 回傳 401/403，但由於 inline 的 `empty($body['success'])` 檢查直接回傳泛化錯誤訊息（"Invalid Page Rules response"），實際的 API 錯誤被吞掉，導致難以 debug。

#### 修復（commit `ae42631`）
1. **auth header 自動注入** — 在 `cf_smart_cache_http_request()`（core.php:571-579）加入：若呼叫方未設定 `headers['Authorization']`，自動從 settings 讀取 token 注入。這影響所有 API 呼叫，且不會重複設定（已有 auth 的呼叫不受影響）
2. **錯誤訊息改善** — 所有 inline `$body['success']` 檢查改為顯示 Cloudflare 原始錯誤訊息 `$body['errors'][0]['message']`

#### 受影響文件
- `cf-smart-cache/includes/core.php:571-579`（auth 注入）、`core.php:970`（page rule error）、`core.php:1016`（zone get error）、`core.php:1045`（zone set error）、`core.php:1169-1171`（dns error）

### [下午 第五階段] 生產級重構 DNS Proxy + 錯誤訊息（6f395df）

#### 深度反思
**三個潛在 bug 在真實 API 測試後才浮現，核心原因全是「假設與現實不符」：**

1. **Auth header 遺漏** — 假設 `cf_smart_cache_http_request()` 會處理認證，但 14 個新函數都沒傳（已修）
2. **`explicit_cache_control` 被當成 Zone 設定** — 此名稱只存在於 Page Rule action，不存在於 `/settings/` API（已修）
3. **DNS `name=` 過濾太嚴格** — Cloudflare API 做 exact match，無法模糊比對子域名（已修）
4. **DNS name 尾綴 dot** — API 回傳 `example.com.` 但我們的 zone_name 是 `example.com`，root filter 永遠不匹配（本次修）

#### 本次修復（commit 6f395df）
1. **分頁支援** — `cf_smart_cache_get_dns_records()` 改為 pagination loop（per_page=100, page=N），處理 >100 筆記錄的 zone
2. **name 正規化** — 新增 `_name_normalized` 欄位（`rtrim('.')`），root filter 雙向 normalize + 支援 `@` 根記錄慣例
3. **apply_dns_proxy 回傳結構化** — `{updated, skipped, errors, detail}` 取代 raw array，handler 可顯示「3 updated, 2 already proxied」
4. **403 權限提示** — Page Rule 403 時追加明確指引：「請在 Cloudflare Dashboard 為 API Token 新增 Page Rules: Edit 權限」
5. **DNS 結果分類** — 無記錄 → `no matching records found`；已全 proxied → `no changes needed`；混合 → `X updated, Y already proxied, Z errors`

### [下午 第六階段] Page Rule edge_cache_ttl=0 被 plan 拒絕（aa23f2c → 當前）

#### 最終根因
Cloudflare API `messages` 陣列揭露：`.settings[1].value: Your plan allows for a minimum TTL of 7200`

Page Rule 的 `edge_cache_ttl=0`（Respect Existing Headers）在 Free/Pro 方案不被接受，最低值 7200s。`edge_cache_ttl` 本身也不需要 — `cache_level=cache_everything` + `explicit_cache_control=on` 已足夠讓 Cloudflare 尊重 origin 的 `Cache-Control` headers。

#### 教訓
硬編碼 API 參數值是脆弱的。應該用 API 查詢 zone 的 plan，再動態決定可用動作。

#### 計劃：Phase 4b — Plan-Aware Configuration
1. **`cf_smart_cache_get_zone_plan()`** — GET /zones/{zone_id} 回傳 plan id，transient 24h
2. **`cf_smart_cache_get_plan_limits($plan)`** — 靜態對照表：max_page_rules、edge_cache_ttl_min
3. **`get_config_status()` 強化** — 加入 plan_limits、page_rules_used、page_rule_available；移除對 rate_limit_cf_plan 設定的依賴
4. **Admin UI** — Zone 行改為「Zone: example.com (FREE) — Page Rules: 0/3」
5. **Apply handler** — Apply 前檢查 slots，不足時跳過 Page Rule 並顯示警告
