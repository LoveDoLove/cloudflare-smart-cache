# 2026-07-05 AI 工作日誌

## 完成的工作

### [上午] 優化 Cloudflare API 請求速率限制（Rate Limiting Overhaul）

#### 設計階段
- **分析現有問題**：`cf_smart_cache_check_rate_limit()` 固定時窗、硬編碼 1000 上限（非 1200）、所有呼叫者忽略回傳值、無退避重試、無 Purge API token bucket 感知、無請求隊列
- **研究 Cloudflare 官方文檔**：確認 Global API limit = 1200/5min，Purge API = token bucket 模型（Free: 5 req/min, bucket 25, max 100 ops/req），支援 `retry-after` header
- **產出完整生產級計劃**：滑動時窗 + Token Bucket + Adaptive Limit + Exponential Backoff with Jitter + Debounced Queue + HTTP Executor Retry + Admin UI Dashboard

#### 實作階段
- **新核心 Governor**：`cf_smart_cache_rate_governor()` — 滑動時窗計數、狀態機（normal/warning/critical/backoff）、自動調適限流（遇 429 降 10%，下限 600）
- **Token Bucket**：`cf_smart_cache_purge_bucket()` — 根據 Cloudflare 方案選擇參數、支援 burst
- **智慧退避**：`cf_smart_cache_backoff_delay()` — exponential 級數 [1,2,4,8,15] + ±20% jitter + `retry-after` header 感知
- **429 Response Handler**：`cf_smart_cache_handle_429_response()` — 調降 adapted_limit、track 連續 429、1 小時無 429 逐漸恢復
- **HTTP Executor**：`cf_smart_cache_http_request()` — 封裝所有 `wp_remote_*` 呼叫、3 次重試、讀 `retry-after` header、限流檢查
- **Debounced Purge Queue**：`cf_smart_cache_enqueue_purge()` + `cf_smart_cache_flush_purge_queue()` — 合併 2 秒內多個 purge 事件、去重、滿 100 URLs 強制 flush
- **修改所有 Caller**：`batch_purge()`、`execute_purge()`、`purge_all_cache()`、`fetch_zones()` — 全部改用 `cf_smart_cache_http_request()`、嚴格檢查限流回傳值
- **修改所有 Hook 端**：`on_status_change()`、`on_delete_post()`、`on_term_change()` — 改用 `cf_smart_cache_enqueue_purge()` 取代直接呼叫
- **Admin UI**：在 `display_cache_status()` 新增 Rate Limit 狀態區塊、新增 settings 欄位（max requests、retries、adaptive、plan type、batch size）
- **修正 Flaws**：`cf_smart_cache_check_rate_limit()` 日誌寫 1200 但判斷 1000 的不一致

### 主要決策
- **選擇滑動時窗而非固定時窗**：避免 5 分鐘邊界瞬間雙倍請求的已知問題
- **Token Bucket 用個別 transient**：與 global rate limit 分離管理，因為 Purge API 有其獨立的 token bucket 模型
- **Debounce queue 用 wp_schedule_single_event**：而非持續 polling，減少不必要的 transient 讀寫
- **保留舊函數 `cf_smart_cache_check_rate_limit()`**：內部改為呼叫新 governor，維持向後相容

## 遇到的問題和解決方案
- **問題**：舊函數 `cf_smart_cache_check_rate_limit()` 被多處呼叫但回傳值被忽略
  - **解決**：全部改寫為 `cf_smart_cache_http_request()`，回傳值統一為 response array 或 WP_Error
- **問題**：`cf_smart_cache_batch_purge()` 已有 sleep(1) 硬編碼在批次之間
  - **解決**：移除硬編碼 sleep，改用 governor 動態決策
